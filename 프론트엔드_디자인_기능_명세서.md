## 1. 디자인 컨셉

### 1.1 방향성
이 화면은 "관제실(Control Room) 디스플레이"를 컨셉으로 잡았다. 관제 담당자가 장시간 모니터를 주시하는 환경이라는 점, 그리고 이상 상황이 발생했을 때 즉시 시각적으로 인지되어야 한다는 요구사항(4.1-1)에서 출발한 선택이다.

### 1.2 디자인 토큰

| 구분 | 값 | 용도 |
|---|---|---|
| 배경 | `#0A0E13` | 전체 배경 (어두운 관제실 환경 가정) |
| 패널 | `#121820` / `#171F29` / `#1D2733` | 카드·사이드바·모달 등 표면 레이어 (밝기 단계별 구분) |
| 테두리 | `#232B35` | 기본 구분선 |
| 텍스트 | `#E8ECF1`(기본) / `#6B7684`(보조) / `#414B58`(비활성) | 정보 위계 구분 |
| 시그널 컬러 | `#22D3EE` (시안) | 강조·선택 상태·인터랙션 포인트 (레이더 신호 컨셉) |
| 정상 상태 | `#2ED573` | 방 상태 = 정상 |

| UI용 폰트 | Inter | 버튼·라벨·본문 등 일반 UI 텍스트 |

### 1.3 시그니처 요소 — 레이더 스윕 모티프
헤더의 원형 아이콘이 실제 레이더처럼 지속적으로 회전(conic-gradient + CSS animation)한다. 이는 이 시스템의 핵심 기술이 60GHz 레이더 센서라는 점을 시각적으로 드러내는 장치이며, 이벤트 발생 시 방 카드에 나타나는 파동형 펄스 애니메이션과 같은 맥락으로 설계했다. 순수 장식이 아니라 "무엇을 감지하는 시스템인가"를 형태로 보여주는 요소다.

### 1.4 레이아웃 구조
```
┌────────────────────────────────────────────────────────────┐
│ 상단바: 시설명 / 전체·층별 뷰 전환 / 병실추가 / 이벤트 시뮬레이션 / 시계 │
├───────────────────────────────────────────┬──────────────────┤
│                                           │  활성 이벤트 패널  │
│   메인 영역 (층 블록 → 라인 → 방 카드)        │  (상황 해제 버튼)  │
│                                           ├──────────────────┤
│                                           │  이벤트 로그       │
└───────────────────────────────────────────┴──────────────────┘
```
- 넓은 모니터 사용을 가정해 좌측 메인 영역 + 우측 고정폭(300px) 사이드 패널의 2단 구조로 설계.
- 실제 요양병원 평면도를 재현하지 않고, 방을 추상적 카드로 표현 (요구사항 4.1-2 반영).

## 2. 데이터 구조 설계 (요구사항 4.1-2, 5.3 대응)

### 2.1 시설 구조 데이터 (병원별 교체 대상)
```js
floorsData = {
  [층번호]: [        // 층
    [ {id, label, status, note}, ... ],   // 라인(row) 1
    [ {id, label, status, note}, ... ],   // 라인(row) 2
    ...
  ]
}
```
- **층 → 라인(row) → 방** 3단 구조로 설계하여, 실제 복도/구역 배치를 그대로 반영 가능
- 병원이 바뀌면 이 객체(또는 이에 대응하는 API 응답)만 교체하면 되고, 프론트 로직·이벤트 처리 로직은 변경되지 않음 → 요구사항의 "메시지 포맷/로직은 병원이 바뀌어도 변하면 안 됨" 조건을 만족하는 구조

### 2.2 방(room) 객체 필드

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string | 방 번호 (예: "101") |
| `label` | string | 화면 표시용 라벨 (예: "101호") |
| `status` | `'ok'` \| `'fall'` \| `'breath'` | 현재 상태 — 서버 이벤트 메시지의 `eventType`/`status`와 매핑 예정 |
| `note` | `{age, name, detail}` \| `null` | 환자 특이사항 (2.3 참고) |

### 2.3 특이사항(note) 객체
```js
note = { age: "78", name: "김OO", detail: "감기, 거동 시 보조 필요" }
```
표시 형식: `{age}세 {name}환자 특이사항: {detail}`
→ 실제 구현 시에는 이벤트 메시지 스키마와 별개로, **환자 정보 테이블/API**로 분리 설계 필요 (5.3 인터페이스 설계 항목에 추가 권장)

### 2.4 이벤트(events) 객체 — 활성 이벤트 패널용
```js
events = [ { roomId, type: 'fall' | 'breath', time: Date } ]
```
서버로부터 실시간 수신 시 이 배열에 push, "상황 해제" 처리 시 해당 항목 제거하는 구조로 그대로 이관 가능.

## 3. 기능 명세

| # | 기능 | 설명 | 관련 요구사항 |
|---|---|---|---|
| F1 | 전체 보기 / 층별 보기 전환 | 상단 뷰 스위치로 전체 층 또는 특정 층만 필터링해서 표시 | 4.1-3 |
| F2 | 병실 추가 | 층 선택(기존 층 또는 신규 층 생성) 후, 텍스트 영역에 줄바꿈=라인, 쉼표=같은 라인 내 방 구분으로 여러 방을 한 번에 등록 | 4.1-2 (구조 데이터 분리) |
| F3 | 방 삭제 | 카드 우측 상단 `✕` 버튼 → 확인 후 삭제, 연결된 활성 이벤트도 함께 정리 | (신규 요청 반영) |
| F4 | 라인 삭제 | 라인 우측의 🗑 버튼 → 확인 후 해당 라인 전체(포함된 방 전부) 삭제, 빈 라인도 삭제 가능 | (신규 요청 반영) |
| F5 | 드래그 앤 드롭 재배치 | 방 카드를 다른 위치(같은 라인/다른 라인/다른 층)로 드래그하여 순서·배치 변경. 라인 끝 빈 공간에 드롭 시 해당 라인 맨 뒤에 추가 | (신규 요청 반영) |
| F6 | 특이사항 등록/조회 | 방 카드 클릭 → 나이/환자명/특이사항 입력 모달. 저장 시 카드에 🗒 아이콘 표시, 마우스 오버 시 툴팁으로 요약 확인 | (신규 요청 반영, 환자 안전 정보관리) |
| F7 | 이벤트 실시간 표시 | 방 상태 변화(정상→낙상/호흡이상) 시 카드 색상·펄스 애니메이션으로 즉시 표시, 우측 패널에 알림 카드 생성 | 4.1-1 |
| F8 | 상황 해제 | 활성 이벤트 패널의 "상황 해제" 버튼 → 방 상태를 정상으로 복귀, 이벤트 로그에 기록 | 4.1-1 |
| F9 | Mock 이벤트 발생기 | 상단 "낙상/호흡이상 시뮬레이션" 버튼으로 실제 하드웨어 없이 임의 방에 이벤트 발생시켜 테스트 | 4.2 |
| F10 | 이벤트 로그 | 최근 발생/해제 이력을 시각과 함께 목록으로 표시 | 6. (이슈/이력 관리 연계) |
| F11 | 실시간 시계 / 서버 연결 상태 표시 | 관제 화면 상단에 상시 노출 | 4.1-5 (관제 화면 기본 요소) |

**F1: 전체 및 층별 카드에 이미지 추가 및 호실, 화장실, 공용공간, 처치실 등 병원 구성 선택 기능 추가**

## 4. 실제 구현 시 반영 필요 사항 (프로토타입 → 실서비스 전환 체크리스트)

1. **데이터 영속화**: 현재 모든 데이터(방 배치, 특이사항, 이벤트)는 브라우저 메모리에만 존재. 실서비스에서는
   - 시설 구조(F2/F3/F4로 변경된 배치) → PostgreSQL에 저장, 변경 시마다 서버에 갱신 요청 필요
   - 환자 특이사항(F6) → 별도 테이블/API로 설계 (이벤트 메시지 스키마와 분리)
   - 이벤트 로그(F10) → DB의 이벤트 로그 테이블과 동기화
2. **실시간 통신 연동**: F7/F9(현재는 클릭으로 흉내낸 부분)를 WebSocket 또는 SSE 수신으로 교체. 여의치 않으면 polling으로 대체 (기획서 5.1 참고).
3. **상태관리 라이브러리 도입**: 방 개수가 최대 100개 이상으로 늘어날 것을 고려해 Context API보다 Zustand 등 경량 라이브러리 검토 (기획서 5.1 권장 사항).
4. **인증/권한 없음 유지**: F1의 뷰 선택은 로그인 대체 기능으로, 권한 검증 없이 단순 선택으로 유지 (4.1-4 스코프 아웃 준수).
5. **반응형/멀티 모니터 대응**: 현재 좌우 2단 레이아웃은 넓은 화면 기준으로 설계됨. 실제 배포 전 다양한 해상도에서의 검증 필요 (4.1-5).

# 요양병원 평면도 기반 스마트 관제 대시보드 구축 방안 제안서

---

## 1. 개요

본 제안서는 60GHz mmWave 레이더 기반 시니어 환자 낙상·호흡이상 감지 시스템을 병원별 층별 평면도 위에 시각화하여 관제 대시보드로 제공하는 서비스를 다수의 요양병원에 확장 적용하기 위한 구축 방안을 다룬다.

서비스의 핵심 전제는 각 병원의 건물 구조 — 층 수, 층별 용도, 호실 배치, 호실 개수 — 가 모두 상이하다는 점이다. 따라서 특정 병원 하나에 맞춘 화면을 만드는 방식으로는 사업 확장이 불가능하며, **병원이 몇 곳으로 늘어나더라도 코드를 재작성하지 않고 데이터 구성만으로 신규 병원을 온보딩할 수 있는 구조**가 반드시 필요하다.

### 1-1. 현실적 어려움

평면도 확보를 자동화된 방식(공공 데이터 스크래핑 등)으로 해결할 수 있는지 검토한 결과, 다음과 같은 제약이 확인되었다.

- 국내에서 건물 평면도(건축물현황도)는 국토교통부의 건축행정시스템 **세움터** 또는 **정부24**를 통해 열람이 가능하나, **평면도·배치도는 건물 소유자 또는 법적 이해관계자만 열람 가능**하며 신분증·임대차 계약서·위임장 등 증빙서류 제출이 요구된다.
- 열람이 승인되더라도 **보안 및 저작권 문제로 인해 캡처·인쇄·저장 기능이 제한**되는 경우가 많아, 열람 자체가 가능하더라도 파일 형태로 확보하는 것은 별도의 절차가 필요하다.
- 건축 연도가 오래된 건물의 경우 **도면이 전산화되지 않아 시스템상 등록 자체가 없는 경우**가 존재하며, 이 경우 관할 구청 건축과를 통한 원본 도면 열람·복사가 유일한 방법이다.

즉, 평면도는 제3자가 임의로 수집할 수 있는 공개 데이터가 아니라 **건물 소유자(병원) 본인이 직접 신청해야 확보되는 자료**이며, 자동 수집 경로는 존재하지 않는다.

### 1-2. 제안 방향

위 제약을 고려할 때, 평면도 확보 자체는 자동화 대상이 아니라 **영업·설치 프로세스에 포함되는 표준 절차**로 정의한다. 대신 다음 두 가지를 자동화·표준화의 대상으로 설정한다.

1. **병원별로 상이한 건물 구조를 코드 수정 없이 반영할 수 있는 데이터 구조 설계**
2. **확보한 도면(형태 무관)을 해당 데이터 구조로 신속하게 변환할 수 있는 온보딩 도구 개발**

이를 통해 신규 병원 추가 시 개발 리소스 투입 없이, 표준화된 절차만으로 서비스를 확장할 수 있는 체계를 구축하고자 한다.

## 2. 문제 정의

| 구분 | 내용 |
|---|---|
| 층 수 | 병원마다 상이하며, 입원실로 사용하는 층도 병원별로 다름 |
| 호실 배치 | 층별 호실 위치, 형태, 동선 구조가 병원마다 상이함 |
| 호실 개수 | 층별 호실 수가 병원마다 다르며 동일 병원 내에서도 층별로 다름 |
| 평면도 확보 | 병원 소유자를 통한 개별 요청이 유일한 합법적 경로이며, 자동 수집 불가 |
| 도면 형태 | CAD 원본, PDF, 스캔 이미지, 사진 등 병원마다 제공 형태가 상이함 |

결과적으로 시스템은 "특정 병원에 맞춘 화면"이 아니라 **임의의 건물 구조를 입력값으로 받아들일 수 있는 범용 구조**로 설계되어야 한다.

## 3. 해결 방안

### 3-1. 데이터 모델 설계

건물 구조를 코드가 아닌 데이터로 표현하여, 병원이 추가될 때마다 데이터베이스에 레코드를 추가하는 것만으로 신규 병원이 반영되도록 설계한다.

```
Hospital (병원)
 └─ Building (동, 필요 시 여러 동)
     └─ Floor (층)
         └─ Room (호실)
             ├─ 좌표 (평면도 위 폴리곤 좌표, 0~1 비율로 정규화)
             ├─ 타입 (입원실 / 간호스테이션 / 화장실 등)
             ├─ 정원 및 침대 수
             └─ Sensor 목록 (매핑된 mmWave 레이더 장비)
```

핵심은 **호실 좌표를 픽셀 단위가 아닌 0~1 사이의 상대 비율로 저장**하는 것이다. 이를 통해 병원마다 평면도 이미지의 해상도·크기가 달라도 대시보드 렌더링 시 자동으로 스케일이 맞춰지며, 층 수·호실 수 차이는 단순히 데이터베이스의 행(row) 개수 차이로 처리되어 코드 변경이 발생하지 않는다.

### 3-2. 온보딩 도구 (도면 → 데이터 변환)

실제로 자동화가 필요한 지점은 "확보한 도면을 위 데이터 구조로 옮기는 작업"이다. 병원으로부터 제공받는 도면의 형태는 아래와 같이 상이할 것으로 예상되며, 형태별로 처리 난이도가 다르다.

| 제공 형태 | 처리 난이도 | 처리 방식 |
|---|---|---|
| CAD 원본 (DWG/DXF) | 낮음 | 벡터 데이터이므로 파싱 라이브러리를 통해 좌표·치수 반자동 추출 가능 |
| PDF (벡터 포함) | 중간 | 벡터 정보 추출은 가능하나 정확도가 낮아 수동 보정이 필요 |
| 스캔 이미지 / 사진 | 높음 | 벡터 정보가 없어 자동 인식이 사실상 불가능, 수동 작업 필요 |

대다수의 요양병원은 CAD 원본을 보유하지 않거나 제공하기 어려운 경우가 많아, 실제로는 스캔 이미지·사진 형태로 도면을 받을 가능성이 높다고 판단된다. 따라서 완전 자동 인식(AI 기반 도면 자동 판독)에 초기 리소스를 투입하기보다, **관리자용 웹 기반 온보딩 에디터**를 우선 구축하는 방식을 제안한다.

**온보딩 에디터 동작 방식**

1. 담당자가 병원 도면 이미지를 업로드
2. 웹 캔버스 상에서 호실 경계를 마우스로 클릭하여 폴리곤 형태로 지정
3. 각 폴리곤에 호실명, 용도, 층 정보를 입력
4. 저장 시 자동으로 0~1 정규화 좌표로 변환되어 데이터베이스에 반영
5. 센서 설치 시, 등록된 호실 목록에서 선택하여 레이더 장비 ID를 매핑

이 방식은 병원 구조가 아무리 상이하더라도 신규 병원 1곳당 약 30분~1시간 내외의 작업만으로 대시보드 반영이 가능하도록 하며, 개발자 개입 없이 비개발 인력(설치 기사 등)도 수행할 수 있는 절차로 표준화한다.

### 3-3. 확장 단계 (중장기)

병원 수가 늘어나고 CAD 원본 데이터가 누적되는 시점에는, 벡터 도면에 한해 자동 방 인식(컴퓨터 비전 기반 폴리곤 자동 추출) 기능을 보조 도구로 추가하는 것을 중장기 과제로 설정한다. 다만 초기 단계에서는 정확도와 개발 리소스 대비 효율을 고려하여 수동 트레이싱 방식의 온보딩 에디터를 우선 구축한다.

---

## 4. 사업적 의의

본 온보딩 에디터는 단순한 내부 도구가 아니라 **서비스의 확장성을 담보하는 핵심 자산**으로 기능한다.

- 병원별 맞춤 개발 없이 표준화된 절차만으로 서비스 확장이 가능해짐에 따라, 영업 단계에서 "도면 제공 시 단기간 내 대시보드 구축 가능"이라는 명확한 경쟁력을 제시할 수 있다.
- 반대로 병원마다 개발 인력이 투입되어야 하는 구조라면, 병원 수 증가에 따라 개발 리소스가 선형적으로 증가하여 사업 확장에 구조적 한계가 발생한다.
- 평면도 확보 절차(도면 요청)를 계약·설치 프로세스에 표준 항목으로 포함시킴으로써, 확보 과정의 불확실성을 사전에 통제할 수 있다.

## 5. 실행 로드맵

| 단계 | 내용 |
|---|---|
| 1 | 데이터베이스 스키마 확정 (Hospital → Building → Floor → Room → Sensor, 좌표 정규화 방식 포함) |
| 2 | 도면 업로드 및 호실 폴리곤 트레이싱 관리자 웹 에디터 개발 |
| 3 | 센서-호실 매핑 UI 개발 (현장 설치 기사가 직접 등록 가능하도록) |
| 4 | 대시보드를 데이터 기반으로 렌더링하도록 구현 (병원별 하드코딩 배제) |
| 5 | 영업 프로세스에 "도면 요청(형태 무관: CAD/PDF/이미지 모두 허용)"을 표준 계약 절차로 명시 |
| 6 (중장기) | CAD 원본 데이터 누적 이후, 벡터 도면 대상 자동 방 인식 기능 추가 검토 |

---

## 6. 결론

평면도 확보는 자동화가 불가능한 영역이므로 영업·계약 프로세스의 일부로 흡수하고, 대신 **병원별로 상이한 건물 구조를 코드 변경 없이 데이터로 수용할 수 있는 구조**와 **도면을 신속하게 데이터로 변환할 수 있는 온보딩 도구**를 구축함으로써 서비스 확장성을 확보하는 것을 핵심 방향으로 제안한다.
